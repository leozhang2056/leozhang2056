### Q11. How do you handle background tasks in Android?

**Key Points**
1. WorkManager → deferrable, guaranteed, survives process death
2. Foreground Service → user-aware long-running tasks
3. Real example: Enterprise Messaging persistent connection

**Script**
> It depends on the nature of the task.
>
> For deferrable work that needs guaranteed execution — syncing data, uploading logs, processing queues — I use WorkManager. It handles process death, system reboots, and battery optimization properly. You define constraints like network availability and let the system schedule it.
>
> For user-visible long-running work — things the user knowingly initiated that need to run continuously — a Foreground Service is the right tool. It shows a persistent notification, which tells the user something is running and prevents the system from killing it.
>
> In the Enterprise Messaging app, we needed a persistent TCP connection to the server for push messages. We used a Foreground Service to maintain that connection, with a heartbeat mechanism and reconnection logic. The NDK layer handled the actual socket in C, but the lifecycle management sat in the service.
>
> I'd avoid plain background Services for anything modern — they get killed aggressively on API 26+ and aren't reliable.

### Q12. How do you implement offline-first architecture in Android?

**Key Points**
1. Room as single source of truth
2. Repository pattern: always read from local, sync to remote async
3. Real example: Smart Factory / Forest Patrol

**Script**
> The core principle is: the local database is the single source of truth. The UI never reads directly from the network.
>
> I use Room as the local store, with a Repository layer that the ViewModel talks to. The Repository handles two things: reading from Room and triggering network sync in the background. When data comes back from the server, it writes to Room, and the UI updates reactively through LiveData or Flow.
>
> For conflict resolution, it depends on the domain. In the Smart Factory app, production records were device-generated and essentially append-only — no conflict. In the Forest Patrol app, rangers could edit inspection records offline and sync later — there we used a last-write-wins strategy with a server-side timestamp, which was sufficient for the use case.
>
> WorkManager handles the sync scheduling so it retries on network availability without draining the battery.

### Q13. How do you optimize Android app performance?

**Key Points**
1. Memory: avoid leaks, use WeakReference carefully, LeakCanary
2. UI: RecyclerView + DiffUtil, avoid overdraw, Systrace
3. Network: batch requests, caching, pagination
4. Native: NDK when Java isn't fast enough (Enterprise Messaging)

**Script**
> I approach it in layers.
>
> For memory, the most common issue is leaking Contexts or Views through static references, inner classes, or callbacks holding references longer than their lifecycle. LeakCanary surfaces these quickly. I also profile heap dumps when there are unexplained GC pressure spikes.
>
> For UI performance, RecyclerView with DiffUtil is the baseline — you never want full dataset refreshes. I use Systrace or the GPU overdraw visualizer to identify rendering bottlenecks. Avoiding transparent layers and unnecessary background draws makes a noticeable difference on mid-range devices.
>
> For network, I batch requests where possible, implement response caching with appropriate cache headers, and paginate large lists instead of fetching everything upfront.
>
> And when Java-layer performance genuinely isn't sufficient — as it wasn't for the <200ms latency requirement in Enterprise Messaging — NDK is the right answer. The native C socket implementation removed the JNI overhead on every message send/receive and gave us the consistency we needed.

### Q14. How do you handle Android app architecture?

**Key Points**
1. MVVM with clean separation of concerns
2. Repository pattern, Use Cases for business logic
3. Jetpack components: ViewModel, LiveData/Flow, Room, Navigation

**Script**
> I follow MVVM as the baseline architecture, using Jetpack components — ViewModel for lifecycle-aware state, LiveData or StateFlow to surface data to the UI, and Room for local persistence.
>
> For larger apps I add a domain layer with Use Cases between the ViewModel and the Repository. This keeps business logic testable in isolation — you can unit test a Use Case without touching the database or the UI.
>
> The Repository is the boundary between the domain and data layers. It abstracts away whether data comes from Room, a REST API, or a cache. The ViewModel doesn't know or care about the source.
>
> In practice, I've found that strict layering pays off most when you're maintaining a system over years. On the Smart Factory app, we had multiple developers working on it over six years — the architecture meant you could change the data source without touching business logic, which saved a lot of pain during refactors.

### Q15. How do you handle Android app security?

**Key Points**
1. Data at rest: EncryptedSharedPreferences, Keystore
2. Data in transit: HTTPS, certificate pinning for high-security apps
3. Code: ProGuard/R8, avoid logging sensitive data
4. Auth: token-based, refresh rotation

**Script**
> A few layers to cover.
>
> For data at rest, I use EncryptedSharedPreferences for sensitive user data like tokens, and Android Keystore for cryptographic keys. Never plain SharedPreferences for anything sensitive.
>
> For data in transit, HTTPS is the baseline. For enterprise apps with high security requirements — like the messaging platform — I add certificate pinning to prevent man-in-the-middle attacks, though you need a plan for certificate rotation.
>
> For authentication, I use JWT with short expiry and a refresh token rotation strategy. The refresh token is stored in EncryptedSharedPreferences. On the server side, we invalidate refresh tokens on logout.
>
> For code protection, R8 with aggressive minification and obfuscation. And critically — no sensitive data in logs, ever. We had a code review rule that flagged Log.d calls that touched anything related to tokens or user data.

### Q17. How do you design a REST API?

**Key Points**
1. Resource-oriented, nouns not verbs
2. Proper HTTP method semantics
3. Versioning strategy
4. Error responses with meaningful codes and messages

**Script**
> I start with the resource model — what are the domain entities and what operations do they support? URIs should be nouns, not verbs: `/orders/{id}` not `/getOrder`.
>
> HTTP methods map to intent: GET for reads, POST for creation, PUT/PATCH for updates, DELETE for removal. PUT replaces the entire resource; PATCH applies partial updates. I use whichever is semantically correct rather than defaulting to POST for everything.
>
> Versioning through the URL path — `/api/v1/...` — is pragmatic and client-friendly. Header-based versioning is cleaner in theory but harder to test and debug.
>
> Error responses matter a lot. A 400 should tell the client exactly what was wrong — field name, constraint, message. A 500 should never leak internals. I use a global exception handler that maps domain exceptions to appropriate HTTP codes and a consistent error envelope.
>
> I also think about idempotency for anything that modifies state — especially important for payment-style or critical business operations where the client might retry on timeout.

### Q18. How do you approach database design and optimization?

**Key Points**
1. Normalization vs denormalization tradeoffs
2. Indexing strategy (cover queries, not all columns)
3. Slow query analysis, EXPLAIN
4. Caching with Redis for hot data

**Script**
> Design starts with understanding the access patterns — what queries will be frequent, what's the read/write ratio, what latency targets exist.
>
> I normalize to at least 3NF to eliminate redundancy, but I'll denormalize selectively when read performance demands it — materialized views or denormalized summary tables for reporting, for example.
>
> For indexing: index the columns in your WHERE, JOIN, and ORDER BY clauses. Composite indexes need to match query patterns — leftmost prefix rule. I don't over-index; every index slows writes. I use EXPLAIN to verify the query planner is actually using the indexes I expect.
>
> For the Smart Factory system we had production tracking tables with hundreds of millions of rows after a few years. We partitioned by date, which kept query times manageable without sharding.
>
> For hot data — user sessions, frequently read config, leaderboard-style aggregates — Redis is the right layer. The messaging platform used Redis heavily for presence tracking and recent message caching.

### Q19. How do you design for high availability?

**Script**
> The goal is eliminating single points of failure.
>
> At the application layer: stateless services behind a load balancer, with health checks and automatic failover. Kubernetes or at minimum a process supervisor to restart crashed instances.
>
> At the database layer: primary-replica replication for reads, with failover capability. For write-critical systems, something like MySQL Group Replication or Galera Cluster.
>
> For the Smart Factory system, we had local deployments at each factory site with edge caching — if the cloud connection dropped, the factory floor kept running on cached data and synced when connectivity restored. That was critical because factory floor operations couldn't wait for a network hiccup.
>
> Monitoring and alerting is the other half of HA. You can't respond to failures you don't know about. We used Prometheus with alerting rules for latency and error rate thresholds.

### Q20. How do you handle microservices challenges — specifically distributed transactions?

**Key Points**
1. Problem: ACID across services is hard
2. Options: 2PC (avoid), Saga pattern (preferred)
3. Real example: Smart Factory order → production allocation

**Script**
> Distributed transactions are one of the hard problems in microservices. Two-phase commit works in theory but it's slow, tightly couples services, and the coordinator is a single point of failure. In practice I avoid it.
>
> For most business workflows, the Saga pattern is the right answer. You break the transaction into a sequence of local service transactions, each of which publishes an event or message. If a step fails, compensating transactions roll back the previous steps.
>
> In the Smart Factory system, the flow was: create order → allocate to factory → schedule production → update inventory. Each of those was a separate service. We used choreography-based Sagas — each service listened for events and reacted, with compensating events if something failed. This kept services loosely coupled.
>
> The tradeoff is eventual consistency — there's a window where the system is in a partially committed state. You design the UX to handle that gracefully, and you build idempotent compensating transactions so retries don't cause duplicate side effects.

### Q22. How did you fine-tune the model in ChatClothes? Why LoRA?

**Key Points**
1. OOTDiffusion baseline → pose alignment wasn't consistent enough
2. LoRA: parameter-efficient, train a fraction of weights, don't overwrite base model
3. Practical: needed to run on Raspberry Pi, full fine-tune was infeasible

**Script**
> The base OOTDiffusion model had good generation quality but inconsistent pose alignment when the garment and the person had very different poses. I wanted to improve that without full retraining — which would have required significant compute and wouldn't have fit the edge deployment constraint anyway.
>
> LoRA — Low-Rank Adaptation — solves this elegantly. Instead of updating all the weights in the attention layers, you add two small low-rank matrices whose product approximates the weight update. You only train those small matrices — typically less than 1% of the total parameters. Training is fast, the result can be merged back into the base model or applied as an adapter, and you don't destroy the general capabilities the original model was trained for.
>
> For my use case, I trained the LoRA adapter on a curated subset of the DressCode dataset with a focus on pose diversity. The result was meaningfully better pose alignment without a significant latency penalty at inference time.

### Q23. How do you approach deploying ML models to constrained hardware?

**Key Points**
1. Model selection: smaller architecture first (YOLO12n not YOLO12x)
2. Quantization: INT8, FP16
3. ONNX for cross-platform runtime
4. Profiling: find the actual bottleneck before optimizing

**Script**
> Start with the constraint: what's the memory budget, what's the acceptable latency, what's the power envelope? That determines how aggressive you need to be with optimization.
>
> My first step is always model selection — choose the smallest architecture that meets the accuracy requirement. For ChatClothes, YOLO12n was the starting point, and I further modified it to YOLO12n-LC by removing components not needed for classification, which reduced both model size and inference time.
>
> Next is quantization. FP32 is the default but FP16 is usually sufficient and cuts memory roughly in half. INT8 goes further but needs careful calibration — you can lose accuracy if the quantization range isn't well-calibrated. On ARM hardware like the Raspberry Pi, FP16 gave good results.
>
> ONNX is valuable for cross-platform deployment — you train in PyTorch, export to ONNX, and run with ONNX Runtime on any target. This also unlocks hardware-specific optimizations through execution providers.
>
> And always profile before you optimize. On the Raspberry Pi the actual bottleneck was the LLM inference, not the diffusion model — which changed where I focused my optimization effort.

### Q24. What is RAG and when would you use it?

**Script**
> RAG — Retrieval-Augmented Generation — is a pattern where you augment an LLM's response with relevant documents retrieved from an external knowledge base, rather than relying solely on the model's parametric knowledge.
>
> The use case is anything where you need the LLM to answer questions about specific, up-to-date, or proprietary information that wasn't in the training data. A company's internal documentation, a legal database, a product knowledge base.
>
> The basic pipeline: user query → embed the query → retrieve the top-k similar chunks from a vector database → inject those chunks into the prompt as context → LLM generates an answer grounded in that context.
>
> The important engineering considerations are chunking strategy — how you split documents matters a lot for retrieval quality — and the retrieval mechanism. Dense retrieval with embeddings is the default. Hybrid search combining dense and sparse (BM25) often works better in practice. Re-ranking the retrieved results with a cross-encoder before passing to the LLM is another common improvement.
>
> I've worked with this architecture in the context of my thesis and in the KB system I've built for my own job search tooling.
